# Epic: Project-owned routing and migration

## Epic Goal
Enable teams to define, manage, validate, preview, migrate and audit HTTP routing inside each project folder so routing is versioned with the project, ownership is clear, and migrations from a central routing config are safe and reversible.

## Related Stories
- Add project-scoped route definition
- Edit project route definition
- Remove project route definition
- View & list project route definitions
- Route precedence & conflict resolution
- Migrate single project routing
- Generate migration report
- Rollback a migration
- Validate route definition on save
- Preview route changes for a project
- Audit & history of route definitions

## User Journey
A developer adds a route file to their project, validates and previews it, then applies it. Platform operators can migrate central routes into project folders, review an autogenerated report, verify behavior with tests and previews, and either finalize or roll back. All changes are auditable and searchable.

---

### Title: Add project-scoped route definition
Story Format: As a project maintainer, I want to add a route definition file to my project's folder, so that routing for my project is defined together with the code.
Description: Project owners declare which host/path maps to their service in a route file stored in the project so the route is versioned and owned by the team.
Acceptance Criteria:
- Given I have write access to a project folder, when I add a route definition file and the change is applied, then requests matching the defined host/path return the expected project response (status and representative content).
- Given other projects or central configuration exist, when I add a project route, then requests for other projects remain unaffected.
- When the route file is added, then the project’s route listing shows the new route with author and timestamp and confirms it is active or queued for activation.
User Value: Developers can own routing alongside code, enabling faster iteration and clearer ownership.

---

### Title: Edit project route definition
Story Format: As a project maintainer, I want to edit a route definition file in my project folder, so that I can change where or how project traffic is routed.
Description: Teams need to update routes to reflect service changes, new paths, or targets.
Acceptance Criteria:
- Given an existing route file, when I change its contents and the change is applied, then requests for matching host/path follow the updated behavior (observable via test requests).
- Given the previous routing behavior, when the edit is applied, then test requests demonstrate the new response and the old response is no longer returned for matching requests.
- When viewing the route, then the change log shows who edited it and when.
User Value: Teams can quickly adapt routing as services evolve without blocking on platform operators.

---

### Title: Remove project route definition
Story Format: As a project maintainer, I want to remove a route definition file from my project, so that traffic is no longer directed by that rule.
Description: Teams must be able to retire routes when services are removed or replaced.
Acceptance Criteria:
- Given a route file exists, when I remove it and the removal is applied, then subsequent requests that previously matched that route no longer reach the previous target and instead follow documented fallback behaviour (e.g., not found or other configured fallback).
- Given routes in other projects, when I remove my route, then other projects’ routes and behavior remain unchanged.
- When removal is completed, then the project route listing and history record the removal with author and timestamp.
User Value: Teams can safely decommission routing rules they no longer need.

---

### Title: View & list project route definitions
Story Format: As a project maintainer or platform operator, I want to list and view route definitions for a project, so that I can audit and verify routing for that project.
Description: Users need a clear view of the project’s active and historical routes and their status.
Acceptance Criteria:
- Given a project with route files, when I request the project’s routes, then I see each route’s host/path, a short target summary, active/inactive status, last-modified time and author.
- When a route is invalid or inactive, then its status is clearly indicated along with a human-readable reason.
- When I open a route entry, then a concise, user-readable summary describes what requests it matches and where they go.
User Value: Easier auditing and debugging of project routing without searching central config.

---

### Title: Route precedence & conflict resolution
Story Format: As a project maintainer, I want clear precedence rules and conflict feedback when project routes overlap central or other project routes, so that routing behaviour is predictable.
Description: When multiple sources define overlapping routes, users must understand which rule will apply and why.
Acceptance Criteria:
- Given overlapping definitions, when a request matches multiple rules, then the observed routing follows a documented precedence and the UI or response indicates which rule applied (demonstrable by test requests).
- When creating or editing a project route that would conflict with an existing route, then the system surfaces a clear conflict warning describing the overlap and the expected outcome.
- When viewing the project route listing, then any conflicts are flagged and include suggested next steps (e.g., review or confirm).
User Value: Predictable routing reduces surprises and support incidents.

---

### Title: Migrate single project routing
Story Format: As a platform operator, I want to migrate a single project’s existing central routes into that project’s folder, so that the project becomes self-contained without changing end-user behaviour.
Description: Operators need a safe way to convert centrally managed routes into project-owned files.
Acceptance Criteria:
- Given central routing entries that map to a project, when migration for that project is executed, then equivalent route definition files appear in the project folder and a migration summary is produced.
- Given traffic previously served by central routes, when migration completes and is enabled, then end-to-end test requests for the project’s hosts/paths return the same status and representative content as before migration.
- When a migration would change behaviour for a route, then the migration flags the difference and requires explicit confirmation before finalizing.
User Value: Enables gradual transition to project-owned routing with minimal user impact.

---

### Title: Generate migration report
Story Format: As a platform operator, I want a human-readable migration report showing old-to-new mappings and differences, so that I can review and approve the migration.
Description: Migration must produce evidence for review, approval and traceability.
Acceptance Criteria:
- When a migration is prepared or run, then a report is generated listing each migrated route, the new project file location, any detected behavioural differences and recommended actions.
- When reviewing the report, then operators can view or export it and use it to approve or halt the migration.
- When differences are detected, then each difference includes short examples of expected request/response behavior.
User Value: Provides confidence and traceability for changes that affect live traffic.

---

### Title: Rollback a migration
Story Format: As a platform operator, I want to rollback a project’s migration, so that I can restore previous routing if the migration introduces problems.
Description: Rollback must restore observable routing behaviour to the pre-migration state.
Acceptance Criteria:
- Given a migration was applied, when I initiate a rollback, then the project’s routing returns to the prior observed behaviour for matching requests (verifiable via end-to-end tests).
- When rollback completes, then files added by the migration are removed or clearly marked inactive and the rollback action is recorded with author and reason.
- When performing rollback, then test requests demonstrate that pre-migration responses are restored.
User Value: Provides a safety net that enables cautious, reversible migration.

---

### Title: Validate route definition on save
Story Format: As a project maintainer, I want route definitions to be validated when I save them, so that I get immediate human-readable feedback and avoid broken routes.
Description: Prevent invalid or inconsistent route definitions from being applied silently.
Acceptance Criteria:
- Given I create or edit a route file, when I save it, then I receive validation results that are either a clear success confirmation or a list of human-readable errors/warnings.
- When validation fails, then the route is not made active and the reason is visible to me so I can fix it.
- When validation succeeds, then the saved route is eligible to be previewed and the save shows author and timestamp.
User Value: Reduces risk of downtime from misconfigured routes.

---

### Title: Preview route changes for a project
Story Format: As a project maintainer, I want to preview how a route change will behave for chosen test requests, so that I can confirm the change before making it live.
Description: Previewing lets teams see expected request/response behavior without impacting live traffic.
Acceptance Criteria:
- Given an un-applied route change, when I request a preview for a specific host/path, then I see a representative response (status and sample body/headers) demonstrating what the change would do.
- When the preview response differs from current live behavior, then the difference is highlighted so I can decide to adjust or apply the change.
- When I accept the preview, then the change can be finalized; when I cancel, no live routing is altered.
User Value: Lowers risk by enabling verification before changes affect users.

---

### Title: Audit & history of route definitions
Story Format: As a platform operator or project maintainer, I want a change history and audit trail for route definitions, so that I can see who changed what and restore prior versions if needed.
Description: Full traceability for compliance, debugging and recovery.
Acceptance Criteria:
- When viewing a route, then a history shows previous versions, authors, timestamps and a short change summary.
- When needed, then a prior version can be restored or its content copied into the project for re-application.
- When migrations or rollbacks occur, then those events appear in the audit trail with associated metadata and a short description.
User Value: Accountability, easier troubleshooting and ability to recover from mistakes.

---

*Quality checks applied: stories avoid implementation specifics, acceptance criteria focus on observable user behavior, and stories are scoped to be deliverable in a single sprint.*
